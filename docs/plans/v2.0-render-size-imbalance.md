## 2. 가설 검증 (H1~H4)

### H1: "모든 chunk가 같은 viewBox 안에 그려져서 큰 chunk 기준 fit → 작은 chunk 상대적으로 더 작게"
**부분 해당, 부정확.** 각 chunk는 별도 `<svg>`. 단일 viewBox 문제는 아니다. 그러나 `scaleRowSvgs()`가 각 SVG를 `containerWidth`에 맞게 **개별 독립 다운스케일**한 뒤, `applyTransform()`이 **하나의 CSS transform**을 inner div에 공유 → 실질 효과는 H1과 동일.

### H2: "chunk별 별도 svg지만 동일 zoom transform 공유 → 한쪽에 맞추면 다른 쪽 무너짐"
**정확히 일치. 주요 원인.**

`scaleRowSvgs()`에서 큰 chunk SVG는 `natW=4000` → `width=containerWidth(≈1280)` (scale ≈ 0.32x), 작은 chunk SVG는 `natW=400` → `width=400` (scale ≈ 1.0x)로 설정. 두 SVG의 시각적 scale이 이미 3~10배 차이. 이후 줌 버튼은 `ST[t].s`를 동일 적용 → 큰 chunk 확대 시 작은 chunk가 화면 밖.

**스모킹 건**: `viewer.html:344-349` — `natW > containerWidth`인 chunk는 강제 축소, 그렇지 않은 chunk는 원본 크기 유지.

### H3: "짧은 라벨 그룹은 좁고 긴 라벨 그룹은 넓게 → 시각적 불균형"
**부분 기여, 직접 원인 아님.** mermaid dagre는 노드 라벨 길이에 따라 viewBox 폭 결정. `/api/v1/users/admin` 같은 긴 경로 chunk는 자연 viewBox 폭이 큼. 이 자연 폭 차이가 `scaleRowSvgs()` 다운스케일 비율 차이를 증폭. H2 선행 조건.

### H4: "classDef에 폰트 사이즈 명시 없음 → mermaid default + zoom out → 서브픽셀 깨짐"
**독립 확인. H2와 직교하는 별도 원인.**

`mermaid-renderer.ts:34` RENDERING_INIT `themeVariables`에 `fontFamily`만 있고 `fontSize` 없음. `:36-43` CLASS_DEFS에도 font-size 미지정. `mermaid.initialize()`의 `flowchart.htmlLabels:false`는 SVG vector text라 zoom 시 래스터화는 OK이나, mermaid 기본 폰트(~14-16px in viewBox)가 `scaleRowSvgs()` 다운스케일로 3x 축소되면 유효 ~5px → 브라우저 anti-aliasing 한계 진입 → 텍스트 깨짐.

`fontFamily 'JetBrains Mono'`는 모노스페이스 — 고정폭 특성상 짧은 라벨도 충분 셀 폭 요구. H3와 반대 방향.

## 3. 사용자 첨부 이미지 매칭

### DB-Screen 탭 — "16 rows"

### A. Chunk 크기 정규화
`buildRenderingDiagram()` `:389-395`, `buildScreenComponentDiagram()` `:562-568`에서 "1 top-level branch = 1 chunk" 정책(v1.1.6) 변경 → route 수 균등화. 예: `/api` 200 routes → 50개씩 4 sub-chunk.

문제: 의미 단위 청크 파괴. `/api`가 여러 chunk로 나뉘면 navigational coherence 손실. v1.1.53 `SINGLE_DIAGRAM_ROUTE_THRESHOLD` 도입 의도(regression 방지)와 충돌.

### B. 노드 폰트/패딩 강제 통일 — 추천(단기)
`mermaid-renderer.ts:34` `themeVariables`에 `fontSize:'16'`(또는 `'18'`) 추가:
```js
themeVariables: { ..., fontSize: '16', ... }
```

매우 낮은 구현 비용. mermaid가 viewBox 단위로 큰 폰트 베이크인 → 3x 다운스케일 후에도 ~5px 이상 유지. 단독 한계: 비율 불균형 자체 미해결. viewBox 폭도 폰트 크기에 비례 증가 → 다운스케일 비율 동일.

부작용: 자연 viewBox 폭 증가 → 넓은 chunk 더 넓어져 H3 약화 가능.

### C. Chunk별 독립 viewBox + 독립 zoom
각 `.row-diagram`에 개별 `ST` 상태 부여, 별도 wheel/drag 이벤트. `scaleRowSvgs()`는 초기 fit only, 이후 사용자 조작은 해당 chunk에만.

위치:
- `viewer.html:336-353` `scaleRowSvgs()` — 초기 크기만
- `viewer.html:644-655` `applyTransform()` — chunk별 개별 transform
- `viewer.html:663-686` 이벤트 핸들러 — `.canvas-wrap` → `.row-diagram` 단위

비용 높음. shared ST 상태 모델 전면 개편. 가치: 각 chunk 독립 확대 가능. 부작용: chunk 간 연결 시각적 단절.

### D. 페이지네이션 — 추천(중기)
`renderChunked()` 결과를 한 번에 모두 삽입하지 않고, 현재 chunk index만 표시 + prev/next.

위치:
- `viewer.html:297-312` `renderChunked()` — chunks 배열 보존, 현재 index render만
- `viewer.html:125-129` `.chunk-nav` 기존 UI 재활용 (현재 "N rows" 표시)
- `activateRowMode()` 대신 `activatePaginationMode()` 신설

비용 중간. 가치 높음 — `fitToView()` 단일 SVG 경로 재활용 → chunk마다 최적 fit 자동.

부작용: 전체 동시 파악 불가. DB 16 rows 시나리오에서 16번 클릭 필요. → **하이브리드**: chunk 수 ≤ 8 row-mode 유지, > 8 페이지네이션.

### E. 줌 레벨 기반 재렌더 — 폐기 수준
mermaid render는 비동기 + 무거움. wheel 이벤트마다 재렌더 불가.

### F. Minimap — 단독 효용 낮음
위치 안내는 되나 가독성 미개선. D와 결합 시 시너지.

## 5. 추천 조합 + Why — **사용자 결정 반영 (2026-05-15)**

**사용자가 옵션 C 채택 — "한눈에 모든 chunk 비교 + chunk별 독립 zoom"**.

이전 안(B+D)은 폐기. 사용자가 첨부 미리보기 본 모양은 **2×2 grid + chunk별 독립 zoom**이므로 실제 구현은 두 변경의 결합:

### C-1: chunk별 독립 zoom 상태
- 각 `.row-diagram`에 개별 `ST` (`scale/x/y`) 상태
- wheel/drag 이벤트 핸들러가 hover된 `.row-diagram`에만 적용
- 기존 shared inner-div transform 폐기

### C-2: grid layout
- 현재 `row-mode`의 `flex-direction:column` (수직 단일 컬럼)을 **CSS Grid 다중 컬럼**으로 변경:
  ```css
  .canvas-wrap.row-mode .inner {
    display: grid;
    grid-template-columns: repeat(auto-fit, minmax(560px, 1fr));
    gap: 16px;
    padding: 16px;
  }
  ```
- 자동 wrap → 화면 폭에 따라 N×M 그리드 형성 (16 rows → 2×8 또는 4×4)

### 부수 개선 (작은 비용으로 큰 효과): B 보조 적용
폰트 floor 상승은 별도 결합 가능:
- `mermaid-renderer.ts:34, 578` themeVariables에 `fontSize:'14'` 추가 (1줄)
- C로 chunk별 fit 적용해도 자연 viewBox 폭이 큰 chunk(컬럼 많은 ER)는 여전히 작아질 수 있으므로 floor 확보

### 위험·미해결

| 항목 | 처리 |
|---|---|
| drag pan 경계 정의 | hover된 chunk에만 drag 적용. chunk 경계 넘어가면 새 chunk로 인터랙션 이동 |
| `fitToView()` chunk별 적용 | 각 `.row-diagram`이 자체 `fitToView` 호출하도록 viewer.html 리팩터 |
| chunk 간 edge 시각화 (있는 경우) | grid layout으로 분리되면 chunk 간 edge가 시각적으로 끊김. **현재 코드는 chunk 분할 시 edge가 같은 chunk 안에서만 의미가 있으므로 영향 적음** — 단 확인 필요 |
| 회귀 차단 | 28-routes/mini-next-app fixture는 chunk 미발생 → `fitToView()` 단일 SVG 경로 유지 (변경 없음) |

### 장기 (참고)
Cytoscape 마이그레이션은 메모리 `project_cytoscape_migration_plan.md`에서 **보류 결정**. 보류 해제 시 chunk 자체가 불필요해질 수 있으나, 현 시점 본 트랙은 mermaid 측에서 완결.
