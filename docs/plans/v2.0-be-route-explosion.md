# Track 2: BE 라우트 급등 (938 routes / 408 tables) 분석

> 작성: 2026-05-15
> 담당 트랙: WINA-APP-BE — 이전 9x → 현재 938로 급증한 원인 진단.

## 1. 어댑터 감지 흐름

`detectStack()` (`packages/llm/src/stack-detector.ts:175`) 실행 순서:

1. 루트 `package.json` deps → JS/TS 프레임워크
2. `requirements.txt` / `pyproject.toml` → Python
3. `pom.xml` OR `build.gradle` OR `build.gradle.kts` 존재 → `'springboot'` 확정 (`stack-detector.ts:107-110, 190-194`)
4. 모노레포 서브앱 fallback
5. 전체 하위 디렉토리 scan fallback

**"WINA-APP-BE"는 프로젝트 이름으로 스택 추론하지 않음.** `pom.xml`/`build.gradle*` 중 하나가 루트에 있으면 무조건 `SpringBootAdapter` 선택. 이후 `adapter.analyze()` (`packages/core/src/adapters/springboot/adapter.ts:19`)에서 5개 파서 병렬 실행.

## 2. 라우트 카운팅 규칙 (Spring Boot)

`parseAnnotations()` (`packages/core/src/adapters/springboot/parsers/annotation-parser.ts:144`):

**진입 조건**: 클래스에 `@RestController` 또는 `@Controller` 존재 (`annotation-parser.ts:23, 173-174`)

**카운팅 수식**:
```
RouteNode 수 = Σ (classPrefixes 길이) × (methodSuffixes 길이)
```
- `classPrefixes`: 클래스 레벨 `@RequestMapping` 경로 배열. 없으면 `['']`
- `methodSuffixes`: 메서드 레벨 어노테이션 경로 배열. 없으면 `['']`

예: `@RequestMapping({"/v1", "/v2"})` + `@GetMapping({"/a", "/b"})` → 2 × 2 = **4 RouteNode** (`annotation-parser.ts:217-244`)

**NodeId 형식**: `route:{relPath}:{urlPath}:{ann.name}` (`annotation-parser.ts:226`)
push 전 dedup 없음 — 결과 배열에 중복 RouteNode가 누적될 수 있음.

## 3. 중복/노이즈 후보 패턴 (5가지)

### 패턴 1 — `src/test` 디렉토리 미제외 (확정적 결함, 가장 유력)

`packages/core/src/adapters/_shared/file-finder.ts:5`:
```ts
JAVA_EXCLUDE_DIRS = {'.git', 'node_modules', 'target', 'build', '.gradle'}
```

`test` 디렉토리가 없다. Spring Boot 표준 `src/test/java`의 `@WebMvcTest` mock 컨트롤러, integration test용 `@RestController` stub이 전부 실 라우트로 집계. 대규모 BE에서 test controller 수가 main의 30~50%에 달하는 경우 흔함.

### 패턴 2 — N-11 배열 prefix 곱셈 (확정적 증폭 요인)

`@RequestMapping({"/api/v1", "/api/v2"})` 배열 지원 이후, API 버저닝 패턴 컨트롤러의 라우트 수가 prefix 배열 길이만큼 즉시 곱해짐 (`annotation-parser.ts:217`). 이전 버전에서 단일 경로로 처리됐다면 N-11 추가가 카운트를 수배로 올렸을 가능성.

### 패턴 3 — 동일 경로 이중 어노테이션 (낮음)

`@GetMapping("/a")` + `@RequestMapping("/a", method=GET)` 둘 다 달린 메서드 → NodeId의 `ann.name`이 달라 별개 RouteNode 2개 생성. 실 코드에서 드물지만 가능성 존재.

### 패턴 4 — `@FeignClient` / `@HttpExchange` 오탐 (이론상 낮음)

파서가 `@RestController` OR `@Controller`를 가진 `class_declaration`만 진입 (`annotation-parser.ts:162, 173-174`) → interface 기반 `@FeignClient`는 이론상 스킵. 단, `@RestController` abstract class를 구현체가 상속 + 메서드 재어노테이션 패턴은 이중 카운트 가능 → **사용자 실측 필요**.

### 패턴 5 — 빌드 결과 복사본 중복 스캔 (낮음)

`target/`/`build/`는 JAVA_EXCLUDE_DIRS에 있으나, `.codesight/` 캐시 또는 비표준 출력(`out/`)은 미제외. 소스 복사본이 다른 디렉토리에 있으면 이중 스캔 → **사용자 실측 필요**.

## 4. 408 테이블 카운팅 분석

테이블 merge 파이프라인 (`adapter.ts:30-39`):
```
JPA entities (name-based Map dedup) → MyBatis 보충 → Flyway DDL 보충 → tableNodes
```

- **JPA**: `tables.push()` 이지만 `adapter.ts:30`의 `new Map(jpaNodes.map(n => [n.name, n]))`에서 이름 기준 Map → 최종 dedup OK. 단 `src/test` 내 `@Entity` 클래스는 테이블명이 다르면 카운트 증가 가능.
- **MyBatis**: `globalRegistry` Map name dedup OK (`mybatis-parser.ts:210-215`). 단 `extractTablesFromSql()`이 SQL `FROM/INTO/UPDATE/JOIN` 뒤 첫 식별자를 모두 테이블로 등록 → 뷰(VIEW), CTE, 서브쿼리 alias도 포함될 수 있음 (`mybatis-parser.ts:23-28`).
- **Flyway**: `tableMap` Map name dedup OK (`flyway-parser.ts:138-166`). `MIGRATION_DIRS` 3개 경로 순서 수집, 동일 SQL 다른 위치 복사본은 `tableMap.has(name)` 체크에서 먼저 온 파일이 이김 → 이중 등록 없음.

408 테이블은 큰 B2B 시스템에서 실존 가능 수치이나, `src/test` `@Entity` + MyBatis SQL alias 오탐이 합쳐질 경우 과잉 집계 가능 → **사용자 실측 필요**.

### 코드 레벨 자명 중복 push 가능성 enumerate

| 시나리오 | 자명 중복 발생? |
|---|---|
| 같은 메서드에 `@GetMapping + @RequestMapping(method=GET)` 동시 | **이론상 발생** (다른 ann.name → 별개 노드). 실 코드에서 드묾 |
