# Track 1: React Router 분석기 미흡 수정 계획

> 작성: 2026-05-15 (feature/v2.0-cytoscape-poc 브랜치)
> 담당 트랙: React Router (Vite + react-router-dom) 분석기. 27개 라우트만 추출되는 문제.

## 1. 현재 동작 (Evidence)

### 1-1. 파일 발견 단계

`parseReactRouterFull` (`packages/core/src/adapters/reactrouter/parsers/route-parser.ts:274`) 및 `parseReactRoutes` (`route-parser.ts:571`) 두 함수 모두 동일한 파일 발견 로직을 공유.

**파일 게이트 조건** (`route-parser.ts:283-289`):
- `createBrowserRouter` / `createHashRouter` / `createMemoryRouter` 문자열 포함 → `routerFiles`
- `<Routes` 문자열 포함 → `jsxRouterFiles`
- **둘 다 없으면 해당 파일은 완전히 무시됨**

`{MobileRoutes}`, `{appRouteElements}` 등의 참조 대상 파일이 `<Routes>`도 `createBrowserRouter`도 포함하지 않는 경우(예: JSX fragment `<><Route.../><Route.../></>` 만 export) → 파일 자체가 탐색 대상에서 제외.

### 1-2. importMap 제한

`route-parser.ts:309-315`, `route-parser.ts:460-464`:
```ts
for (const decl of sf.getImportDeclarations()) {
  const di = decl.getDefaultImport()      // default import만
  if (di !== undefined) imap.set(...)
}
```
**named import (`import { appRouteElements }`, `import { MobileRoutes }`) 는 완전히 무시됨.**

### 1-3. JSX child 루프의 expression 처리 공백

`extractJsxRouteChildren` (`route-parser.ts:215-266`):
```ts
// route-parser.ts:226-237
if (child.isKind(SyntaxKind.JsxElement)) {          // <Route ...>...</Route>
  tagName = ...
} else if (child.isKind(SyntaxKind.JsxSelfClosingElement)) {  // <Route ... />
  tagName = ...
}
// JsxExpression ({변수명}) 분기 없음
// → tagName = undefined → line 239: if (tagName !== 'Route') continue
// → 아무런 경고 없이 silent skip
```

`SyntaxKind.JsxExpression` (`{appRouteElements}`, `{MobileRoutes}`, `{MobileLoginRoute}`) 분기 부재. `tagName`이 `undefined`로 남고 `continue`로 조용히 버려진다.

### 1-4. 기존 2-pass sub-router 시스템의 범위

`route-parser.ts:456-487`, `route-parser.ts:668-688`의 2-pass 시스템은 다음 두 조건 **모두** 만족해야 작동:
1. 컴포넌트가 `element={<X/>}` prop 안에 있어야 함
2. 대상 파일이 `<Routes` 문자열 포함

사용자 케이스의 `{appRouteElements}`, `{MobileRoutes}` 등은 **element prop이 아니라 JSX child expression** → 조건 1 불성립. 2-pass 시스템이 존재해도 이 버그 해결 불가.

### 1-5. 잡는 패턴 / 못 잡는 패턴

| 패턴 | 예시 | 현재 처리 |
|---|---|---|
| literal `<Route path="..."/>` | `<Route path="/login" element={...}/>` | 정상 |
| nested `<Route>` 자식 path | `<Route path="/users"><Route path=":id"/></Route>` | 정상 |
| `<Route index>` | `<Route index element={<X/>}/>` | 정상 (parentPath 재사용) |
| path-less layout wrapper | `<Route element={<Layout/>}>` | 현재 라우트 노드 생성(`routePath = parentPath || '/'`) — 의미론적으로 잘못된 중복 가능성 |
| `path="*"` catch-all | `<Route path="*"/>` | 정상 (`/*` 변환) |
| JSX child expression — 같은 파일 변수 | `{appRouteElements}` | **silent skip** |
| JSX child expression — named import | `{MobileRoutes}` | **silent skip + 파일도 미발견** |
| `createBrowserRouter` 배열 변수 | `createBrowserRouter(routes)` (변수명) | 정상 (같은 파일 VariableDeclaration만) |
| named import 참조 | `import { MobileRoutes } from './mobile'` | **importMap 미등록** |
| 데이터 배열 `.map()` → JSX | `appRoutes.map(r => <Route path={r.path}/>)` | **skip** |

## 2. 사용자 시나리오 매핑

27개로 줄어든 원인 = 다음 4가지 갭의 복합 작용:

**Gap 1 — JSX child expression 완전 무시** (`route-parser.ts:226-239`)

사용자 Router.tsx의:
- `{MobileLoginRoute}` ← Gap 1
- `{appRouteElements}` ← Gap 1
- `{MobileRoutes}` ← Gap 1

세 줄 모두 `SyntaxKind.JsxExpression` 노드. child 루프에 해당 분기가 없어 조용히 건너뜀.

**Gap 2 — named import를 importMap이 추적하지 않음** (`route-parser.ts:311`)

`MobileRoutes`, `MobileLoginRoute`가 별도 파일에서 named export로 나왔다면:
```tsx
import { MobileRoutes } from './mobile-routes'
import { MobileLoginRoute } from './mobile-login-route'
```
`getDefaultImport()`만 수집 → 모듈 경로 알 수 없음.

**Gap 3 — 파일 발견 게이트** (`route-parser.ts:287`)

참조 대상 파일(`mobile-routes.tsx`)이 내부적으로 `<><Route/><Route/></>` (JSX fragment) 형태로만 export하면 `<Routes` 문자열 없음 → `jsxRouterFiles` 불포함 → ts-morph project에 미추가.

**Gap 4 — `.map()` 데이터 배열** (`route-parser.ts:59-131`, `extractRoutesFromArray`)

`const appRouteElements = appRoutes.map(r => <Route path={r.path} element={r.element}/>)`: 배열 원소가 object literal이 아닌 JSX이므로 `extractRoutesFromArray`의 `ObjectLiteralExpression` 분기와 미스매치. 별도 처리 경로 없음.

## 3. 수정 방향 (옵션 비교)

### 옵션 A: 같은 파일 변수 추적만 (~50% 회복)

`extractJsxRouteChildren`에서 `JsxExpression` child를 만나면 → 식별자를 얻어 → **같은 소스파일** 내의 `VariableDeclaration` initializer를 찾아 → `ArrayLiteralExpression`이면 재귀 수집, `CallExpression(.map)`이면 데이터 배열 처리.

- 대상: `{appRouteElements}` (동일 파일에서 `appRoutes.map(...)` 결과를 const에 담는 패턴)
- 미해결: 다른 파일에서 import한 `{MobileRoutes}`, `{MobileLoginRoute}`
- 변경 LOC: ~55 LOC

### 옵션 B: 같은 파일 + 단일 파일 named/default import 추적 (~80% 회복) — 추천

옵션 A + 다음:
1. importMap을 named import까지 확장 (default + named 모두 수집)
2. `JsxExpression` 식별자가 importMap에 있으면 → 해당 모듈 경로 파일을 ts-morph project에 추가
3. 그 파일의 top-level export expression에서 `<Route>` 수집 (fragment `<>...</>` unwrap 포함)
4. 파일 발견 게이트는 변경하지 않고, resolution 시점에 `project.addSourceFileAtPath` 직접 호출

- 대상: `{MobileRoutes}`, `{MobileLoginRoute}` (named import, 다른 파일)
- 주의: 재귀 깊이 제한 필요 (1-hop only)
- 변경 LOC: ~118 LOC

### 옵션 C: 풀 import 그래프 추적 (~95% 회복, 복잡도 높음)

import chain 재귀 탐색. 별도 파일 그래프 빌더(DAG) + 순환 참조 방지.

- 변경 LOC: ~250+ LOC, 새 모듈 분리 필요
- 복잡도: 높음. 파싱 시간 선형 증가. 현재 아키텍처와 미스매치.

### 추천안: 옵션 B (단계적으로 A → B)

**이유:**
1. 사용자 Router.tsx에서 `{appRouteElements}`가 대량 라우트를 포함할 가능성 높고(most routes), `{MobileRoutes}` 등도 named import가 일반적.
2. 옵션 A만으로는 named import 패턴 미해결.
3. 옵션 B는 1-hop 파일 레벨 추적으로 끝나며, 풀 DAG(C) 복잡도 없이 현실적 수렴.
4. **1-hop으로 제한**하면 Less is More 원칙 부합 — 2-hop 이상은 confidence 하락 → inferred 처리 부적절.

## 4. 구체 변경 (파일/함수 단위)

### 4-1. importMap 확장

**대상**: `route-parser.ts:309-315`, `route-parser.ts:460-464`

```ts
// 수정 (default + named 모두)
const di = decl.getDefaultImport()
if (di !== undefined) imap.set(di.getText(), decl.getModuleSpecifierValue())
for (const ni of decl.getNamedImports()) {
  imap.set(ni.getName(), decl.getModuleSpecifierValue())
}
```

예상: 각 위치 +4 LOC, 총 +8 LOC

### 4-2. 신규 헬퍼: `resolveIdentifierToJsxChildren`

**위치**: `route-parser.ts` (extractJsxRouteChildren 위, ~line 213)

```ts
function resolveIdentifierToJsxChildren(
  identifierName: string,
  sourceFile: import('ts-morph').SourceFile,
  project: import('ts-morph').Project,
  importMap: Map<string, string>,
  routerDir: string,
  parentPath: string,
): JsxRouteRaw[]
```

**로직**:
1. `sourceFile`의 VariableDeclarations에서 `identifierName` 찾기
   - 있으면: initializer가 `ArrayLiteralExpression` → JSX Route 원소 수집 (Case A 동일파일)
   - initializer가 `CallExpression`에 `.map` → 배열 인수를 `extractRoutesFromArray`로 시도 (Case A 데이터배열)
2. VariableDeclaration 없으면: `importMap.get(identifierName)` → 모듈 경로 취득
   - `undefined`면: unresolved로 기록
   - 있으면: 파일 resolution → `project.addSourceFileAtPath` (Case B named import)
   - 추가된 파일에서 `export default`, `export const identifierName` initializer에서 Route 수집
3. **재귀 불가**: 1-hop만 추적

예상 LOC: ~65 LOC

### 4-3. `extractJsxRouteChildren`에 JsxExpression 분기 추가

**위치**: `route-parser.ts:226` 이후, 기존 `JsxElement` / `JsxSelfClosingElement` 분기 다음

```ts
} else if (child.isKind(SyntaxKind.JsxExpression)) {
  const expr = child.asKindOrThrow(SyntaxKind.JsxExpression).getExpression()
  if (expr?.isKind(SyntaxKind.Identifier)) {
    const resolved = resolveIdentifierToJsxChildren(
      expr.getText(), sourceFile, project, importMap, routerDir, parentPath
    )
    results.push(...resolved)
  }
  continue
}
```

**시그니처 변경**:
```ts
function extractJsxRouteChildren(
  children: JsxChild[],
  parentPath: string,
  ctx: {
    sourceFile: SourceFile
    project: Project
    importMap: Map<string, string>
    routerDir: string
  } | null,
): JsxRouteRaw[]
```

기존 호출부에서 `ctx: null` 전달 시 expression skip (하위 호환).

예상: 헬퍼 연결 ~15 LOC + 시그니처/호출부 수정 ~10 LOC

### 4-4. 미해결 참조 진단

`unresolvedExpressions: string[]` 누적 → adapter 결과 또는 stderr에 진단:
```
[react-router] 3 route references could not be resolved: appRouteElements, MobileRoutes, MobileLoginRoute
```

Less is More: 불확실한 `unknown:appRouteElements` 노드는 생성하지 않음. 진단 메시지만 표면화.

## 6. Less is More / Evidence-First 부합성

### Inferred 노드 생성 여부: **생성하지 않음**

`unknown:appRouteElements` 같은 placeholder 노드 미생성.

**이유**:
- `provenance.line`은 expression 위치로 확정 가능하나 `path` 필드가 완전 미지수
- `confidence: 'inferred'` + `inferenceChain`에 담을 추론 체인이 "identifier name 발견"뿐 → 정보량 0
- 렌더러가 `path: ''` 노드를 받으면 오히려 노이즈

**대신**: `unresolvedExpressions` 진단 출력.

### provenance / inferenceChain

| 케이스 | confidence | inferenceChain |
|---|---|---|
| Case A (same-file const) | `'verified'` | — |
| Case B (1-hop import) | `'verified'` | — |
| Case A `.map()` 데이터배열 | `'inferred'` | `['appRoutes 배열의 path 프로퍼티를 정적 평가']` |

`.map(r => <Route path={r.path}.../>)` 패턴은 `r.path` 실제 값이 런타임 데이터에 의존 → `'inferred'` 적절.
