# Track 4: 트리 노드 X/Y 축 구조 변경 계획

> 작성: 2026-05-15
> 담당 트랙: "depth별 X축 표시, depth 바뀌면 Y축 줄넘김" — mermaid 레이아웃 구조 변경.

## 1. 현재 트리 빌드/렌더 흐름 (file:line)

### 1-1. 데이터 계층 (`packages/renderer/src/url-grouper.ts`)

`groupRoutesByUrl` (L108) → `groupRoutesRecursive` (L38) 순서로 RouteNode[] 를 NestedGroup 트리로 변환.

- LCP(Longest Common Prefix) 세그먼트 기반 재귀 클러스터링
- 결과: `{ groupKey: "/template", routes: [...], children: [{ groupKey: "/template/grid-tree", ... }, ...] }`
- minGroupSize=3 미만 그룹은 재귀 중단 + children=[] 평탄화 (L96)

### 1-2. 렌더 계층 (`packages/renderer/src/mermaid-renderer.ts`)

**Tab1 (Rendering Diagram)**
- `buildRenderingDiagram` (L381) → `buildNestedSubgraphLines` (L117)
- 최상위 `graph TD` (L267) — TD가 전체 레이아웃 방향 결정
- `buildNestedSubgraphLines`: NestedGroup 트리 재귀 순회 + `subgraph ... end` 중첩 emit
- leaf subgraph 노드 수가 NODES_PER_INNER_ROW=5 초과 시 → `emitInnerRowSubgraphs` (L221) invisible inner row + `direction LR` (L238)

**Tab2 (Screen-Component)**
- `buildScreenComponentDiagram` (L510) → `renderScreenSection` (L339) → `buildScreenSubgraphLines` (L275)
- 최상위 `graph TB` (L359)
- subgraph suffix: `_S` (예: `TEMPLATE_S`), comp subgraph: `_C` (예: `TEMPLATE_S_C`)
- 사용자가 제시한 `subgraph TEMPLATE_S[/template]` 코드는 이 경로 — **Tab2가 문제 다이어그램**

**청킹 게이트 (L390~396, L562~571)**
- `branchingGroups.length > GROUPS_PER_ROW(5)` AND `routeNodes.length > SINGLE_DIAGRAM_ROUTE_THRESHOLD(100)`
- 1 top-level branch = 1 chunk → `joinChunks` (wrap-fallback.ts:112) `%%--CHUNK--%%` 구분자

### 핵심 함수

```
groupRoutesByUrl          url-grouper.ts:108   RouteNode[] → NestedGroup[]
groupRoutesRecursive      url-grouper.ts:38    LCP 기반 재귀 클러스터
buildNestedSubgraphLines  mermaid-renderer.ts:117   NestedGroup → mermaid subgraph (Tab1)
buildScreenSubgraphLines  mermaid-renderer.ts:275   NestedGroup → mermaid subgraph (Tab2)
emitInnerRowSubgraphs     mermaid-renderer.ts:221   leaf 노드 > 5 시 invisible row 분할
findBranchingGroups       mermaid-renderer.ts:250   단일 transit 노드 건너뜀
```

## 2. 사용자 요구 재정의 — **within-group 해석 확정 (2026-05-15)**

사용자 답변: **"within-group (부모 안에서 자식 가로)"** 채택.

### within-group 해석

| 사용자 용어 | mermaid 대응 |
|---|---|
| "depth별 X축 나열" | **한 부모 subgraph 안에서** 자식들끼리만 가로 정렬 (LR row 래퍼) |
| "depth 바뀌면 Y 줄넘김" | 한 부모 안에서 자식이 GROUPS_PER_ROW(5)개 넘으면 다음 row 래퍼로. 손주는 자식 안에서 다시 가로 |

NestedGroup tree 계층 유지, **형제만 가로**. `feedback_render_chunked_path_nested.md` 원칙 준수.

### ASCII Mockup — `/template` 7자식

**현재** (각 자식이 독립 subgraph → mermaid TD로 수직 스택):
```
[/template]
    |
[/template/grid-tree]  [/template/grid-save]  [/template/form]
    |                        |                      |
[route]                  [route]                [route]
    |
[/template/grid-tree/detail]
    |
[route]
```

**사용자 원하는 모양** (depth=1 가로, depth=2 그 아래 가로):
```
Row (depth=1):  /grid-tree  |  /grid-save  |  /form  |  /table  | ...
                    ↓                          ↓
Row (depth=2):  /grid-tree/detail  |  /grid-tree/config
```

## 3. mermaid 11.x 가능 여부 — Tab2 코드 실증 (2026-05-15)

공식 문서: `subgraph` 내부 `direction` 키워드 지원 (v11.x). **제한**: "If any of a subgraph's nodes are linked to the outside, subgraph direction will be ignored."

### Tab1 (Rendering)
- routes 간 edge **없음** (`buildRouteRowDiagram` 출력에는 노드만)
- 외부 연결 제한 미해당 → `direction LR` 정상 작동 예상

### Tab2 (Screen-Component) — edge emit 확인됨
`packages/renderer/src/mermaid-renderer.ts:313`:
```ts
allEdges.push(`  ${sanitizeId(r.id)} ${edgeArrow(edge)} ${sanitizeId(compId)}`)
```
route 노드(`r.id`)가 외부 component(`compId`)와 `-->` edge 연결. `:370`에서 모든 edge를 subgraph 외부에 emit.

→ **route 노드를 row 래퍼 subgraph로 묶으면, 그 subgraph 안 route 노드가 외부 comp와 연결됨 → mermaid 제한 발동 → row 래퍼의 `direction LR` 무시**

### 결론 — Tab1/Tab2 결정적 차이

| 탭 | direction LR 적용 효과 |
|---|---|
| Tab1 | ✅ 작동 예상 (외부 edge 없음) |
| Tab2 | ⚠️ **부분 작동 가능성**: row 래퍼 묶음 자체는 시각적 구분 제공, 그러나 mermaid가 LR 방향을 무시할 수 있음 → 자식 subgraph가 여전히 수직 스택될 수 있음 |

**사용자 사전 고지 필요**: 사용자 첨부 두 번째 스크린샷(BE WINA, 4 rows)은 Tab2 — comp edge 존재. 옵션 A 확장이 Tab2에서 기대만큼 작동하지 않을 가능성 있음. 1차 Tab1 검증 후 Tab2 실측 필수.

## 4. 옵션 비교

### 옵션 A: 각 NestedGroup subgraph에 `direction LR` 추가

```
subgraph TEMPLATE_G["📄 /template"]
  direction LR
  subgraph TEMPLATE_GRID_TREE_G[...] ... end
  subgraph TEMPLATE_GRID_SAVE_G[...] ... end
end
```

장점: NestedGroup tree 보존. 코드 변경 1줄.
단점:
- mermaid 제한: Tab2에서 route 노드가 comp와 연결되면 direction LR 무시
- 자동 레이아웃이라 "depth=2가 depth=1 아래 정확히 줄넘김" 보장 불가
- 부모 `graph TD`가 최상위 방향을 TD로 강제

NestedGroup 원칙 충돌: 없음.

### 옵션 B: depth별 row subgraph 평탄화

```
graph TD
  subgraph DEPTH1_R[" "]
    direction LR
    TEMPLATE_G  BLOG_G  API_G
  end
  subgraph DEPTH2_R[" "]
    direction LR
    TEMPLATE_GRID_TREE_G  TEMPLATE_GRID_SAVE_G  ...
  end
```

장점: 사용자 요구 가장 직접 구현.
단점:
- 부모-자식 관계가 subgraph 중첩 → edge 연결로 바뀜 → edge 교차 폭발
- **`feedback_render_chunked_path_nested.md` 명시 금지 패턴** (`collectNestedRoutes flat` 구조와 동일)
- depth가 다른 그룹들이 같은 depth row에 섞이는 의미 단위 손실

NestedGroup 원칙 충돌: **직접 위반. 채택 불가.**

## 6. 추천안 + 코드 변경 지점

### 추천: 옵션 A 확장 — Tab1 한정 검증 후 Tab2 판단

**근거**:
1. NestedGroup tree 원칙 위반 없음
2. 코드 변경 최소
3. Tab1은 renders edge 없으므로 direction LR이 외부 연결 제한에 미걸림
4. Tab2는 mermaid 제한 적용 가능성 높음 → Tab1 검증 후 결정

### Tab1 변경 — `emitInnerRowSubgraphs` 확장 (유력 대안)

현재 `emitInnerRowSubgraphs` (L221)는 leaf 노드 5개 초과 시 invisible row subgraph + `direction LR` 사용. 이를 **group의 children을 "row"로 묶는 방식**으로 확장:

`buildNestedSubgraphLines` (L117~136)에서 `group.children`을 GROUPS_PER_ROW(5)개씩 invisible row subgraph로 묶고 내부 `direction LR`:

```
subgraph TEMPLATE_G["📄 /template"]
  subgraph TEMPLATE_ROW0 [" "]
    direction LR
    subgraph TEMPLATE_GRID_TREE_G[...] ... end
    subgraph TEMPLATE_GRID_SAVE_G[...] ... end
    subgraph TEMPLATE_FORM_G[...]     ... end
    subgraph TEMPLATE_TABLE_G[...]    ... end
    subgraph TEMPLATE_LIST_G[...]     ... end
  end
  subgraph TEMPLATE_ROW1 [" "]
    direction LR
    subgraph TEMPLATE_CONFIG_G[...]   ... end
    subgraph TEMPLATE_EXPORT_G[...]   ... end
  end
end
```

기존 `emitInnerRowSubgraphs`가 leaf route 노드에 적용하는 패턴을 **자식 subgraph 그룹**에도 적용. `buildNestedSubgraphLines` 내부에서 `group.children`을 5개씩 행 묶음으로 래핑 ~15-20 LOC 추가.

## 7. 위험 — NestedGroup 원칙

`feedback_render_chunked_path_nested.md`:
> "chunked 출력 경로는 NestedGroup tree 구조 보존 필수. collectNestedRoutes flat·top-level Map 금지."

| 변경 | 원칙 위배 |
|---|---|
| buildNestedSubgraphLines에 row 래퍼 subgraph 추가 | **위배 없음** — tree 구조 유지, 시각적 묶음만 추가 |
| group.children flat 변환 후 depth-row 생성(B) | **위배** — 금지 패턴 |
| leaf node invisible inner row 유지 | **위배 없음** — 기존 emitInnerRowSubgraphs 동일 |

실질 위험은 mermaid **"외부 연결 시 direction 무시"** 제한. Tab2에서 renders edge가 row subgraph 경계 넘어가면 direction LR 무력화 → 효과 없는 변경.

## 결론 요약

Track-4 사용자 요구("depth별 X, depth 바뀌면 Y")는 mermaid에서 **부분 구현 가능**.

- mermaid는 `direction LR` 지원, 단 **외부 edge 연결 시 무시**.
- Tab1 (route 전용, edge 없음): children을 GROUPS_PER_ROW 단위 invisible row + direction LR로 **효과 기대 가능**.
- 옵션 B(depth별 평탄화): 금지 패턴 → **불가**.
- 옵션 C(cytoscape): 보류 결정 → **spike 실패 확정 전 진입 불가**.
- 추천: **옵션 A 확장** — `buildNestedSubgraphLines`에서 자식 subgraph 5개씩 invisible row + `direction LR`. Tab1 먼저 검증 후 Tab2 적용 결정.
- Track-4는 활성 spike의 5번째 평가 항목으로 병행 편입. spike 기존 4 후보 타임박스 보호.
