## 1a. Tab1 vs Tab2 레이어 불일치 — 설계 분기

### 실측 (합성 fixture `/tmp/rr-sub`, URL `/partner/matMgmt/deco` 외 2건)

**Tab1** (`buildRenderingDiagram` → `buildNestedSubgraphLines`, URL 기반):
```
REACT
  📄 /partner
    📄 /matMgmt          ← matMgmt가 별도 중첩 subgraph 레이어
      deco · CSR
      spec · CSR
    📄 /ordMgmt
      plan · CSR
```

**Tab2** (`buildFeDomainLayeredScreenDiagram` → `buildPkgTree`, `pages/` 폴더 기반):
```
📁 src/pages/partner
  matMgmt (:::pkg 트리노드) --> deco leaf, spec leaf
  ordMgmt (:::pkg 트리노드) --> plan leaf
```

### 근본원인 (코드)

- Tab1: `mermaid-renderer.ts:116` `groupRoutesByUrl(routeNodes)` → **route.path**의 URL 세그먼트로 중첩 그룹 생성. `/partner/matMgmt/...`는 `partner` subgraph 안에 `matMgmt` subgraph를 중첩(`nested.ts:37` `buildNestedSubgraphLines` 재귀).
- Tab2: `fe/tab2-domain.ts:19` `pagesSegments(filePath)` → **컴포넌트 filePath**의 `pages/` 이후 폴더로 트리 생성.

두 탭이 **다른 좌표계(URL vs 파일폴더)**를 쓴다. 실 repo처럼 URL과 폴더가 일치하면 깊이는 비슷해지지만, 표현 방식이 다르다:
- Tab1은 매 세그먼트를 **subgraph(중첩 클러스터, Y-stack)** 로 → matMgmt가 "별도 레이어"로 보임
- Tab2는 중간 폴더를 **`:::pkg` 트리노드 + visible `-->` 엣지**로 → 레이어가 아닌 노드

→ "정상 분리한 Case도 있고 별도 레이어로 분리한 흔적도 있다"는 관찰은 정확. URL 깊이에 따라 Tab1이 `findBranchingGroups`로 단일자식 transit를 collapse(`/partner`만 있으면 `/matMgmt`로 내려감)하기도, 다자식이면 중첩 유지하기도 하기 때문(`layout.ts:54`).

### 판정: 의도된 동작이나 **일관성 결함**. 단, 1b 해소 시 함께 사라지는 종속 문제

Tab1이 URL 전 세그먼트를 미세하게 중첩 레이어링하는 것 자체가 Tab1의 목적(아키텍처 개관)에 과하다. matMgmt 같은 도메인 세분화는 Tab2의 역할이다. 1b 해결 방향(아래)에서 **Tab1을 top-level 도메인 요약으로 전환**하면 이 분기도 동시에 해소된다.

## 1b. Tab1 프레임워크·BE 연계 누락 — 명백한 결함

### 실측 (합성 fixture `/tmp/rr-big`, top-level 도메인 7개)

7개 도메인 입력 시 Tab1 출력:
```
graph LR
  subgraph PARTNER_G["📄 /partner"] ... end
  subgraph AGENCY_G ... end
  ... (chunk 1/2, chunk 2/2)
```
→ **BROWSER/ROUTER/REACT wrapper 전부 없음. API LAYER(axios) 데이터레이어 없음.** URL 그룹만 남음.

대조 — 도메인 5개(`mini-react-router-domain-app`) Tab1:
```
subgraph BROWSER → ROUTER → REACT → (route groups) → end
subgraph DATALAYER["🔌 API LAYER"] → BROWSER -.->|axios| API_GATEWAY
```
→ wrapper·데이터레이어 정상.

### 근본원인 (코드)

`mermaid-renderer.ts:123`:
```ts
if (routeNodes.length > SINGLE_DIAGRAM_ROUTE_THRESHOLD || branchingGroups.length > GROUPS_PER_ROW) {
  const chunks = splitGroupsByNodeBound(branchingGroups, CHUNK_ROUTE_BUDGET, GROUPS_PER_ROW)
  return joinChunks(chunks.map(gs => buildRouteRowDiagram(gs)))   // ← 여기서 wrapper·데이터레이어 미생성
}
```
`buildRouteRowDiagram`(`nested.ts:89`)은 `[RENDERING_INIT, 'graph LR', CLASS_DEFS]` + `buildNestedSubgraphLines`만 emit한다. **FW_CONFIGS wrapper(line 148~)와 데이터레이어(line 168~) 분기는 chunked 경로에서 절대 실행되지 않는다.**

- 게이트 조건 `branchingGroups.length > GROUPS_PER_ROW(5)`는 **v1.2.51 C2**에서 추가됨(이전엔 `routeNodes > 100`만). 실 repo는 도메인 6개 이상이 흔하므로 거의 항상 발동.
- 표준 위반: `FE-DIAGRAM-STANDARD.md` **R-T1.1(인프라 wrapper)·R-T1.5(외부 분기)** 는 Tab1의 필수 계약인데, chunked 경로가 둘 다 폐기. R-T1.7(chunking)과 R-T1.1/R-T1.5 사이에 미해결 모순이 있었고, C2 게이트가 그 모순을 일상적으로 노출시킴.

### 왜 구조적 딜레마인가

wrapper는 **하나의** 외곽 클러스터인데, chunked 출력은 **여러 개의 독립 mermaid 다이어그램**(viewer가 grid 배치)이다. 한 wrapper가 여러 다이어그램을 가로지를 수 없다. 즉 "전 라우트를 wrapper 안에 나열" + "노드 폭증 시 분할"은 동시 만족 불가.

### 판정: 결함. 해결 방향 = Tab1의 표현 단위 재정의

Tab1의 목적은 **"어떤 인프라/엔진이 렌더하고 어디로 연결되는가"의 개관**(표준 §1)이지 전 라우트 enumeration이 아니다. 개관은 본질적으로 O(도메인 수)로 작다.

**제안(표준 R-T1.2/R-T1.7 amendment 필요)**:
- Tab1 = `BROWSER → ROUTER → REACT` wrapper + **top-level 도메인 요약 박스**(도메인명 + 라우트 수 배지, 예 `partner · 24 routes`) + R-T1.5 외부 분기.
- 전 라우트 leaf·matMgmt급 하위 세분화는 Tab2로 위임 → 1a 동시 해소.
- 도메인 수만큼만 노드 → 항상 단일 다이어그램 → chunking 불필요 → wrapper 항상 유지.
- 효과: Tab1=아키텍처 개관 / Tab2=화면-컴포넌트 상세 / Tab3=데이터흐름의 계층성 복원.

> 이 제안은 표준 R-T1.2("ENGINE 안 라우트 nested subgraph")·R-T1.7(chunking) 변경을 수반하므로, 구현 전 `FE-DIAGRAM-STANDARD.md` amendment 선행 필요. R-T1.1·R-T1.5와는 오히려 정합(둘을 항상 보장).

## 2. Y축 연결선 과대·불균일 — 분할 판정

### 핵심 사실 (코드)

- FE 다이어그램은 `RENDERING_INIT`(`constants.ts:4`) 사용 — **flowchart.nodeSpacing/rankSpacing 설정이 전혀 없음** → mermaid 기본값(rankSpacing≈50).
- BE만 `BE_RENDERING_INIT`(`constants.ts:9`) 사용 — `rankSpacing:8, nodeSpacing:25`.

### Tab2(도메인) — 수정 가능 ✅

Tab2 도메인 출력 구조: 평탄한 `HDR_PAGES` subgraph 1개 + `pkg --> leaf` **visible 엣지** + 중첩 subgraph 없음. 이 구조는 dagre 일반 rank 레이아웃이라 `rankSpacing` 옵션이 정상 적용된다(**BE Tab2 DI 체인이 동일 구조에서 rankSpacing 8로 조밀화된 in-repo 선례**).
- "레이어별 길이 상이" 원인: 도메인 chunk마다 트리 깊이가 다름(partner 2단 vs agency 1단, 실측 확인). dagre 글로벌 rank + 멀티라인 leaf 라벨(`📄 파일명`)이 노드 높이를 키워 rank 간격이 벌어짐. **rankSpacing 축소로 기본 간격을 대폭 표준화 가능** (깊이 차이로 인한 절대 길이 편차는 잔존하나 체감 크게 개선).

### Tab1(인프라 wrapper) — 현 구조상 통제 불가 ⚠️ (구조적)

Tab1은 **nested subgraph(BROWSER>ROUTER>REACT) + 외부 edge incoming(`frontendRef -.-> API`) + 내부 `~~~` chain** 조합이다. 이는 내가 v1.2.41에서 직접 실측해 기록한 **"spacing 옵션 전면 무시" 케이스와 정확히 일치**한다:

> `feedback_mermaid_nested_subgraph_spacing.md`: 외부 edge incoming + 내부 chain subgraph는 init directive·`mermaid.initialize`의 nodeSpacing/rankSpacing/padding 모두 미적용. rankSpacing 50→8→1 모두 변화 없음(실측 확정).

또한 표준 §1 v1.1·R-T1.2는 nested 자식 Y-stack을 **mermaid v11 구조적 한계로 이미 수용**한 상태다. 즉 Tab1 내부 Y축 길이는 "설정 누락"이 아니라 "현 구조에서 통제 불가".

- 구조적 우회만 가능: (a) `~~~` → visible 엣지 전환(BE tab1 패턴, `be/tab1.ts` `parentId --> leafId`), 또는 (b) wrapper subgraph 제거. 단 wrapper는 Tab1의 존재 이유라 (b)는 불가.
- **그러나 1b 재설계(도메인 요약)를 적용하면 Tab1 내부 노드 수가 급감 → Y-stack 자체가 짧아져 이 문제도 부수적으로 완화된다.** 별도 spacing hack보다 재설계가 정공법.

### 판정 종합
- **Tab2**: 수정 가능 — FE init에 compact flowchart spacing 도입(BE 선례). **단, 출고 전 webview 실측으로 조밀화 확인 필수**(과거 Y축 오진단 사례 `feedback_mermaid_y_axis_limit.md` 재발 방지).
- **Tab1**: 현 구조상 spacing 옵션으로 통제 불가(실측 기반). 진짜 레버는 1b 도메인 요약 재설계.
